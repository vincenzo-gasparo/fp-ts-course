This is the capstone lesson. After 14 days of learning individual tools, this lesson puts them together in a complete, realistic example — the kind of code you would actually write in a production application. A user profile service that fetches data, validates it, handles errors, and transforms the result — built entirely with the patterns you have learned. Read it as a reference for how all the pieces connect.

## Introduction

Over the past 14 days you have learned the core fp-ts modules one at a time. Today brings them together. There is no new module to introduce — instead, you will see how `Option`, `Either`, `TaskEither`, and `pipe` compose into a complete, realistic application feature.

The scenario is a **user profile service**. When a client requests a user's profile page, the service must:

1. **Validate** the incoming request data (using `Either` for typed validation errors)
2. **Look up optional preferences** (using `Option` for nullable/absent data)
3. **Fetch external profile data** from an API (using `TaskEither` for async + error handling)
4. **Combine and transform** the results (using `pipe` + `map`/`chain`)
5. **Return** a structured result or a typed error

This mirrors real production code: validation at the boundary, safe nullables in the domain, async I/O for external calls, and typed errors at every step.

## The Complete Example

```typescript
import { pipe } from 'fp-ts/function'
import * as O from 'fp-ts/Option'
import * as E from 'fp-ts/Either'
import * as TE from 'fp-ts/TaskEither'

// --- Domain types ---

type UserId = number

type RequestData = {
  userId: unknown  // raw input — shape is unknown until validated
  includeAvatar?: boolean
}

type UserPreferences = {
  displayName: string
  theme: 'light' | 'dark'
  avatarUrl: string | null
}

type ExternalProfile = {
  userId: number
  bio: string
  followers: number
  createdAt: string
}

type ProfilePage = {
  userId:      UserId
  displayName: string
  bio:         string
  followers:   number
  theme:       'light' | 'dark'
  avatarUrl:   string | null
}

type AppError =
  | { _tag: 'ValidationError'; field: string; message: string }
  | { _tag: 'NetworkError';    message: string }
  | { _tag: 'NotFound';        resource: string }
  | { _tag: 'Unauthorized' }

// --- Step 1: Validate the incoming request ---
// Either for validation — invalid input is a typed error, not an exception

const validateRequest = (data: RequestData): E.Either<AppError, UserId> => {
  const id = Number(data.userId)
  if (isNaN(id) || id <= 0) {
    return E.left({
      _tag: 'ValidationError',
      field: 'userId',
      message: `Expected a positive integer, got: ${String(data.userId)}`,
    })
  }
  return E.right(id as UserId)
}

// --- Step 2: Load optional user preferences ---
// Option because missing preferences is expected behavior, not an error

declare const preferencesStore: Map<UserId, UserPreferences>

const loadPreferences = (userId: UserId): O.Option<UserPreferences> =>
  O.fromNullable(preferencesStore.get(userId))

const getDisplayName = (prefs: O.Option<UserPreferences>): string =>
  pipe(
    prefs,
    O.map(p => p.displayName),
    O.getOrElse(() => `User #${Math.random()}`) // fallback if no prefs
  )

const getTheme = (prefs: O.Option<UserPreferences>): 'light' | 'dark' =>
  pipe(
    prefs,
    O.map(p => p.theme),
    O.getOrElse((): 'light' | 'dark' => 'light') // default theme
  )

const getAvatarUrl = (prefs: O.Option<UserPreferences>, include: boolean): string | null =>
  include
    ? pipe(prefs, O.chain(p => O.fromNullable(p.avatarUrl)), O.toNullable)
    : null

// --- Step 3: Fetch external profile data ---
// TaskEither because network I/O is async and can fail

declare const httpClient: {
  get: <T>(url: string) => Promise<T>
}

const fetchExternalProfile = (userId: UserId): TE.TaskEither<AppError, ExternalProfile> =>
  TE.tryCatch(
    async () => {
      const profile = await httpClient.get<ExternalProfile>(
        `https://api.example.com/profiles/${userId}`
      )
      return profile
    },
    (err): AppError => {
      if (err instanceof Response && err.status === 404) {
        return { _tag: 'NotFound', resource: `profile for user ${userId}` }
      }
      if (err instanceof Response && err.status === 401) {
        return { _tag: 'Unauthorized' }
      }
      return { _tag: 'NetworkError', message: String(err) }
    }
  )

// --- Step 4: Combine everything into the final result ---

const buildProfile = (
  userId:  UserId,
  prefs:   O.Option<UserPreferences>,
  profile: ExternalProfile,
  request: RequestData
): ProfilePage => ({
  userId,
  displayName: getDisplayName(prefs),
  bio:         profile.bio,
  followers:   profile.followers,
  theme:       getTheme(prefs),
  avatarUrl:   getAvatarUrl(prefs, request.includeAvatar ?? false),
})

// --- Step 5: The complete pipeline ---

const getProfilePage = (
  request: RequestData
): TE.TaskEither<AppError, ProfilePage> =>
  pipe(
    // Step 1: Validate — Either<AppError, UserId>
    validateRequest(request),

    // Lift from Either to TaskEither so we can chain async steps
    TE.fromEither,

    // Step 2: Load preferences (sync, cannot fail — lift with TE.of)
    TE.chain(userId => {
      const prefs = loadPreferences(userId)

      // Step 3: Fetch external profile (async, can fail)
      return pipe(
        fetchExternalProfile(userId),

        // Step 4: Combine userId + prefs + profile into ProfilePage
        TE.map(profile => buildProfile(userId, prefs, profile, request))
      )
    })
  )

// --- Step 5: Execute at the program edge ---

const renderResult = async (request: RequestData): Promise<string> => {
  const result = await getProfilePage(request)()

  return E.fold(
    (err: AppError): string => {
      switch (err._tag) {
        case 'ValidationError':
          return `400 Bad Request: ${err.field} — ${err.message}`
        case 'NotFound':
          return `404 Not Found: ${err.resource}`
        case 'Unauthorized':
          return `401 Unauthorized`
        case 'NetworkError':
          return `502 Bad Gateway: ${err.message}`
      }
    },
    (page: ProfilePage): string =>
      `Profile for ${page.displayName}: ${page.bio} (${page.followers} followers)`
  )(result)
}
```

## Patterns Recap

Over 15 days you have learned — and just applied — the five core fp-ts patterns:

**Pattern 1 — pipe for left-to-right composition**
Every pipeline in this example uses `pipe`. It threads a value through a sequence of transformations without intermediate variables, making the data flow explicit and readable.

**Pattern 2 — Option for nullable domain values**
`O.fromNullable`, `O.map`, `O.getOrElse`, `O.chain` — Option models the difference between "absent" and "an error". Missing preferences are not failures; they are expected. Use `null | undefined` as input, use `Option` inside your domain.

**Pattern 3 — Either for typed validation and domain errors**
`E.left` / `E.right`, `E.chain`, `E.fold` — Either makes error handling a first-class concern encoded in the type. Instead of throwing an exception from `validateRequest`, we return `E.left` with a structured error object. Callers handle it in `E.fold` with exhaustive switch logic.

**Pattern 4 — TaskEither for async + error handling**
`TE.tryCatch`, `TE.fromEither`, `TE.chain`, `TE.map` — TaskEither is the workhorse for I/O. It lifts async operations into the same Either-based error model, letting you compose validation, database calls, and API calls in one pipeline without mixing async/await with try/catch.

**Pattern 5 — Execute at the edge**
The fp-ts style keeps your domain logic as pure descriptions of what to do. All actual execution (providing environments, calling `()`, awaiting Promises, pattern-matching on the result) happens at the program's edge — in handlers, controllers, or main functions. This makes the domain logic easily testable in isolation.

## When to Use This Approach

fp-ts is a significant investment. Here is an honest guide to when it is worth it:

**Bring fp-ts in when:**
- Your team works on a medium-to-large TypeScript codebase and correctness matters (finance, health, infrastructure)
- Error handling is currently inconsistent — some functions throw, some return `null`, some return error codes
- You want to eliminate a class of null reference errors at compile time
- The team is willing to invest 1-2 weeks in learning the patterns (Days 1-15 is that investment)
- You have complex async workflows where the error propagation is hard to follow

**Simpler alternatives may be better when:**
- The codebase is small or the feature is a prototype — the learning curve is real
- The team is mostly new to TypeScript — master TypeScript first
- Your async operations are mostly one-off fetch calls — plain async/await is excellent for that
- You need to onboard many junior developers quickly — the cognitive overhead is high initially

fp-ts is not a silver bullet. It is a tool for teams that have decided typed functional programming's benefits outweigh its costs. After 15 days, you can make that decision with full knowledge of what the library offers.

## Key Takeaways

- `Option` models **presence or absence** of a value — use it for nullable domain data, not for errors
- `Either` models **success or typed failure** — use it for validation, decoding, and any operation with a meaningful error value
- `TaskEither` models **async + fallible** — lift async operations with `TE.tryCatch`, compose with `TE.chain`
- `pipe` threads values through transformations left-to-right — the foundation of every fp-ts pipeline
- Execute at the edge: build pure descriptions of what to do, then provide environments and call `()` / `await` at the boundary
- fp-ts shines on correctness-critical, medium-to-large TypeScript codebases — evaluate the team's readiness before adopting

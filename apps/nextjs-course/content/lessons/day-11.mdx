In the previous lesson you learned that Server Components can be `async` functions. This lesson is about what that unlocks: a much simpler and more powerful way to get data into your pages.

## The Old Way vs the Next.js Way

If you've fetched data in a standard React app, you've written something like this:

```tsx
// The classic React pattern — client-side fetching
'use client'

import { useState, useEffect } from 'react'

interface Post {
  id: number
  title: string
}

export default function PostsPage() {
  const [posts, setPosts] = useState<Post[]>([])
  const [loading, setLoading] = useState(true)
  const [error, setError] = useState<string | null>(null)

  useEffect(() => {
    fetch('https://jsonplaceholder.typicode.com/posts?_limit=5')
      .then(res => res.json())
      .then(data => {
        setPosts(data)
        setLoading(false)
      })
      .catch(() => setError('Failed to load posts'))
  }, [])

  if (loading) return <p>Loading...</p>
  if (error) return <p>Error: {error}</p>

  return (
    <ul>
      {posts.map(post => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}
```

This works, but it has real costs: users see a loading spinner, the fetch runs in the browser after the page loads, and all this boilerplate ships as JavaScript to every visitor.

Here is the same page as a Next.js Server Component:

```tsx
// app/posts/page.tsx — Server Component
interface Post {
  id: number
  title: string
}

export default async function PostsPage() {
  const response = await fetch('https://jsonplaceholder.typicode.com/posts?_limit=5')
  const posts: Post[] = await response.json()

  return (
    <ul>
      {posts.map(post => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}
```

No `useEffect`. No loading state. No error state boilerplate. No `'use client'`. The component `await`s the data directly, and the browser receives a complete HTML list — no JavaScript required to render the content.

## Typing Your Fetch Responses

TypeScript can't verify what an API actually returns, but you can declare your intent and catch mistakes early. A clean pattern is to put your fetch logic in a typed helper function:

```tsx
interface User {
  id: number
  name: string
  email: string
  company: { name: string }
}

async function fetchUsers(): Promise<User[]> {
  const res = await fetch('https://jsonplaceholder.typicode.com/users')

  if (!res.ok) {
    // Throwing here will trigger Next.js's error boundary
    throw new Error(`Failed to fetch users: ${res.status}`)
  }

  return res.json() as Promise<User[]>
}

export default async function UsersPage() {
  const users = await fetchUsers()

  return (
    <main>
      <h1>Users</h1>
      <ul>
        {users.map(user => (
          <li key={user.id}>
            <strong>{user.name}</strong> — {user.email}
            <br />
            <span>{user.company.name}</span>
          </li>
        ))}
      </ul>
    </main>
  )
}
```

The `if (!res.ok)` check is important. `fetch` does not throw on HTTP error status codes (404, 500, etc.) — it only throws on network failures. Checking `res.ok` and throwing manually means Next.js can catch the error and show an error page instead of rendering empty or broken data.

## Fetching Multiple Resources in Parallel

When a page needs data from multiple sources, avoid awaiting them one after another. Sequential awaits mean each request waits for the previous one to finish:

```tsx
// SLOW — total time = time(fetchUser) + time(fetchPosts) + time(fetchStats)
export default async function DashboardPage() {
  const user = await fetchUser()
  const posts = await fetchPosts()
  const stats = await fetchStats()

  return <Dashboard user={user} posts={posts} stats={stats} />
}
```

If each fetch takes 200ms, this page takes 600ms to build. Use `Promise.all` instead:

```tsx
// FAST — all three fetches run simultaneously
// Total time ≈ the slowest of the three
export default async function DashboardPage() {
  const [user, posts, stats] = await Promise.all([
    fetchUser(),
    fetchPosts(),
    fetchStats(),
  ])

  return <Dashboard user={user} posts={posts} stats={stats} />
}
```

`Promise.all` starts all the fetches at the same time. The `await` waits until all of them are done. If the three fetches take 200ms, 150ms, and 300ms respectively, `Promise.all` finishes in about 300ms — not 650ms.

Use `Promise.all` whenever the fetches are independent (one doesn't need the result of another).

## Data Fetching with Dynamic Routes

Dynamic routes — pages like `/products/[id]` — also need to fetch data. With static export, you must tell Next.js which IDs exist by exporting `generateStaticParams`. The page then fetches the data for each ID at build time.

```tsx
// app/products/[id]/page.tsx
interface Product {
  id: number
  title: string
  description: string
  price: number
}

async function fetchProduct(id: string): Promise<Product | null> {
  const res = await fetch(`https://fakestoreapi.com/products/${id}`)
  if (!res.ok) return null
  return res.json() as Promise<Product>
}

// Tell Next.js which IDs to pre-render
export async function generateStaticParams() {
  const res = await fetch('https://fakestoreapi.com/products')
  const products: Product[] = await res.json()
  return products.map(p => ({ id: String(p.id) }))
}

// Next.js calls this once per ID returned by generateStaticParams
export default async function ProductPage({
  params,
}: {
  params: Promise<{ id: string }>
}) {
  const { id } = await params
  const product = await fetchProduct(id)

  if (!product) {
    // Triggers the nearest not-found.tsx
    return <p>Product not found.</p>
  }

  return (
    <article>
      <h1>{product.title}</h1>
      <p>{product.description}</p>
      <p>
        <strong>${product.price.toFixed(2)}</strong>
      </p>
    </article>
  )
}
```

`generateStaticParams` fetches the full list of products to discover all valid IDs. Next.js then calls the page component once for each ID, fetching and baking in each product's data. The result is one HTML file per product, each with the data already embedded.

Note that in Next.js 15, `params` is a `Promise` — you must `await` it before accessing properties.

## When to Fetch on the Client

Server Component fetching is ideal for data that's known at build time. But some data depends on what the user does — search queries, filters, form results. For that, you still use `useEffect` in a Client Component.

```tsx
// app/components/search-results.tsx
'use client'

import { useState, useEffect } from 'react'

interface Result {
  id: number
  title: string
}

export function SearchResults({ query }: { query: string }) {
  const [results, setResults] = useState<Result[]>([])
  const [loading, setLoading] = useState(false)

  useEffect(() => {
    if (!query.trim()) {
      setResults([])
      return
    }

    setLoading(true)

    fetch(`https://jsonplaceholder.typicode.com/posts?title_like=${encodeURIComponent(query)}`)
      .then(res => res.json())
      .then((data: Result[]) => {
        setResults(data)
        setLoading(false)
      })
  }, [query])  // Re-runs whenever query changes

  if (loading) return <p>Searching...</p>
  if (results.length === 0) return <p>No results for "{query}".</p>

  return (
    <ul>
      {results.map(result => (
        <li key={result.id}>{result.title}</li>
      ))}
    </ul>
  )
}
```

The rule: if the data is known at build time, fetch in a Server Component. If the data depends on user input or interaction, fetch in a Client Component with `useEffect`.

## Static Export and Build-Time Fetching

When your `next.config.mjs` includes `output: 'export'`, there is no server runtime. Your app is a folder of HTML, CSS, and JavaScript files — like a classic static website.

All fetch calls in Server Components run during `next build`. The data is frozen into the HTML at that moment. If an API returns different data tomorrow, your site will not reflect it until you rebuild and redeploy.

This trade-off is worth it for many kinds of content:

- Blog posts and documentation (content changes infrequently, and you rebuild when it does)
- Product listings (acceptable to rebuild when the catalog changes)
- Course content (like this course — lessons don't change minute to minute)

For data that must always be live — stock prices, social media feeds, user-specific dashboards — you'd need a server runtime (Vercel's platform supports this) or client-side fetching with `useEffect`.

## Putting It Together: A Complete Example

Here is a page that combines server-side fetching for the initial content with a client-side interactive component:

```tsx
// app/blog/page.tsx — Server Component
import { LikeButton } from '../components/like-button'  // 'use client'

interface Post {
  id: number
  title: string
  body: string
}

async function fetchPosts(): Promise<Post[]> {
  const res = await fetch('https://jsonplaceholder.typicode.com/posts?_limit=3')
  if (!res.ok) throw new Error('Failed to load posts')
  return res.json() as Promise<Post[]>
}

export default async function BlogPage() {
  const posts = await fetchPosts()  // Runs at build time

  return (
    <main>
      <h1>Blog</h1>
      {posts.map(post => (
        <article key={post.id}>
          <h2>{post.title}</h2>
          <p>{post.body}</p>
          <LikeButton postId={post.id} />  {/* Client Component — interactive */}
        </article>
      ))}
    </main>
  )
}
```

The posts load instantly from the pre-built HTML. The `LikeButton` hydrates in the browser and can manage its own interactive state with `useState`.

## Key Takeaways

- Server Components can be `async` functions — you `await` data directly, with no `useEffect` or loading state required.
- Data fetched in Server Components is baked into the HTML at build time (for static exports). The browser never sees the fetch call.
- Always check `res.ok` after a fetch and throw an error on failure — `fetch` does not throw automatically on 4xx or 5xx responses.
- Use `Promise.all` to fetch multiple independent resources in parallel, not sequential `await` calls.
- Dynamic routes with static export must export `generateStaticParams` to tell Next.js which paths to pre-render.
- Use client-side `useEffect` fetching when data depends on user input or interaction that cannot be known at build time.
- With `output: 'export'`, all server-side fetches happen at build time — the site reflects data from the last build, not the current moment.

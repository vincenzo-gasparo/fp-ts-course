React components must be **pure** during rendering. That means when React calls your component function, it should only compute and return JSX — no network requests, no timers, no reading from the browser. But real applications need all of those things. `useEffect` is the official escape hatch: a place to run code that reaches outside React after the render is complete.

Think of it this way: React finishes updating the DOM, then taps you on the shoulder and says "okay, now you can talk to the outside world."

## What Belongs in an Effect

Before writing any `useEffect`, it helps to know what qualifies as a side effect:

- Fetching data from an API
- Setting up a WebSocket connection or event listener
- Reading from or writing to `localStorage`
- Starting or clearing a timer
- Integrating a third-party library that manipulates the DOM directly

If your code needs to do any of these things, `useEffect` is the right place for it.

## Basic Structure

```tsx
import { useEffect } from 'react'

useEffect(() => {
  // Setup: this runs after React renders

  return () => {
    // Cleanup: runs before the next setup, or when the component unmounts
  }
}, [dependency1, dependency2])  // Only re-run when these values change
```

There are three parts: the setup function, an optional cleanup function you return from setup, and the dependency array. Each plays a distinct role.

## The Three Dependency Array Patterns

The second argument to `useEffect` controls when the effect re-runs.

```tsx
// 1. No array — runs after EVERY render (rarely useful)
useEffect(() => {
  console.log('rendered')
})

// 2. Empty array — runs once, after the first render only
useEffect(() => {
  console.log('component mounted')
}, [])

// 3. With dependencies — re-runs when those values change
useEffect(() => {
  fetchUser(userId)
}, [userId])
```

The empty array pattern is the most common for initial data loading. The array-with-dependencies pattern is how you react to changing props or state — for example, re-fetching when a selected item ID changes.

## Fetching Data

Data fetching is the most frequent reason to reach for `useEffect`. Here is a realistic, production-quality example:

```tsx
import { useState, useEffect } from 'react'

interface Post {
  id: number
  title: string
  body: string
}

function PostDetail({ postId }: { postId: number }) {
  const [post, setPost] = useState<Post | null>(null)
  const [loading, setLoading] = useState(true)
  const [error, setError] = useState<string | null>(null)

  useEffect(() => {
    let cancelled = false  // guards against stale updates

    setLoading(true)
    setError(null)

    fetch(`https://jsonplaceholder.typicode.com/posts/${postId}`)
      .then(res => res.json())
      .then(data => {
        if (!cancelled) {
          setPost(data)
          setLoading(false)
        }
      })
      .catch(err => {
        if (!cancelled) {
          setError(err.message)
          setLoading(false)
        }
      })

    return () => {
      cancelled = true  // cleanup: ignore result if postId changes
    }
  }, [postId])

  if (loading) return <p>Loading...</p>
  if (error) return <p>Error: {error}</p>
  if (!post) return null

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.body}</p>
    </article>
  )
}
```

### The `cancelled` Flag Explained

Notice the `let cancelled = false` pattern. Here is why it matters.

Imagine a user clicks through a list of posts quickly — post 1, then post 2, then post 3. Each click changes `postId`, which triggers the effect to run again. Three fetch requests are now in-flight simultaneously.

Without the `cancelled` flag, whichever response arrives last will "win" and set the displayed post — even if it was for post 1, which the user already moved past. The screen would show the wrong data.

With the flag: when the effect re-runs (because `postId` changed), the cleanup function from the previous run sets `cancelled = true`. Any response that arrives after that is silently ignored. Only the most recent fetch can update state.

## Cleanup — Why It Matters

The cleanup function prevents resource leaks. Here are two common cases:

```tsx
// Timer that must be stopped
useEffect(() => {
  const id = setInterval(() => setCount(c => c + 1), 1000)
  return () => clearInterval(id)  // cleanup: stop the timer
}, [])

// Event listener that must be removed
useEffect(() => {
  function handleResize() {
    setWindowWidth(window.innerWidth)
  }
  window.addEventListener('resize', handleResize)
  return () => window.removeEventListener('resize', handleResize)  // cleanup
}, [])
```

Without cleanup, the interval keeps ticking and `handleResize` keeps firing even after the component is removed from the screen. These functions hold references back into React state, keeping memory alive that should be freed. Over time — or with components that mount and unmount frequently — this adds up.

The rule is: if your effect allocates a resource, the cleanup function deallocates it.

## React Strict Mode Runs Effects Twice

You may notice during development that your effects appear to run twice. This is intentional. React's Strict Mode deliberately does the sequence: mount → unmount → mount. It does this to verify that your cleanup function fully reverses your setup.

If your app behaves incorrectly on the second mount, your cleanup is incomplete. This is a feature, not a bug — it catches problems in development before they reach production.

## When NOT to Use useEffect

`useEffect` is powerful, but it is also the most misused hook. Two common mistakes:

### Deriving Data

```tsx
// WRONG — unnecessary effect and extra render cycle
useEffect(() => {
  setFullName(`${firstName} ${lastName}`)
}, [firstName, lastName])

// CORRECT — compute directly during render
const fullName = `${firstName} ${lastName}`
```

If a value can be computed from existing state or props, compute it during render. No effect needed. The effect version causes React to render once, run the effect, update state, and render again — double the work for no benefit.

### Handling User Events

```tsx
// WRONG — effects are not for responding to user actions
useEffect(() => {
  if (submitted) {
    sendEmail()
  }
}, [submitted])

// CORRECT — call directly in the event handler
function handleSubmit() {
  sendEmail()
}
```

If something should happen because the user clicked a button or submitted a form, put that logic in the event handler. Effects are for synchronizing with external systems, not for handling interactions.

A useful mental test: can you describe your effect as "keep X synchronized with Y"? If yes, it probably belongs in an effect. If it's more like "do Z when the user does W", it belongs in an event handler.

## Key Takeaways

- `useEffect` runs after React renders — it is the right place for side effects like data fetching, timers, and event listeners
- The dependency array controls when the effect re-runs: no array means every render, `[]` means once on mount, `[value]` means when that value changes
- Always return a cleanup function when your effect allocates a resource — this prevents memory leaks when the component unmounts or the effect re-runs
- Use the `cancelled` flag pattern when fetching data that depends on a prop or state value, to avoid stale responses overwriting newer ones
- React Strict Mode runs effects twice in development on purpose — treat it as a test that your cleanup works correctly
- Do not use `useEffect` for derived values or event responses — those belong in the render body and event handlers respectively

You know how to apply a single transformation to a value inside a box (that is `map`). But what if the transformation itself is inside a box? Or what if you want to combine two independent boxed values — say, two separate API results — into one? `Apply` and `Applicative` solve this. Think of it as running several independent operations in parallel and combining their results, as opposed to chaining operations where each one depends on the previous.

## Introduction

On Day 3 you learned that `Either`'s `chain` short-circuits on the first error. On Day 5 you learned that `TaskEither`'s `chain` sequences operations one at a time. Both of these are the right choice when operations are **dependent** — when step 2 needs the output of step 1. But what about **independent** operations?

Consider loading a user's dashboard: you need to fetch their profile, their recent activity, and their notification count. These three operations do not depend on each other — you could run them all simultaneously. And for form validation: you want to show ALL errors at once, not just the first one. Stopping at the first invalid field frustrates users.

This is what the `Apply` type class enables. `Apply` is the abstraction behind functions like `sequenceS` and `sequenceT` from `fp-ts/Apply`. It lets you combine **independent** effectful computations — multiple `Option` values, multiple `Either` values, or multiple `TaskEither` operations — and collect all results (or all errors) in a single structured result.

The relationship between Apply, Applicative, and Monad:

- **Functor** (`map`): apply a pure function to a value inside an effect
- **Apply** (`ap`, `sequenceS`, `sequenceT`): combine multiple independent effects
- **Monad** (`chain`): sequence dependent effects where each step uses the previous result

In practice you will rarely call `ap` directly. You will use `sequenceS` (for combining named records) and `sequenceT` (for combining tuples) with the appropriate Apply instance.

## Core API

### sequenceS — combining named independent effects

```typescript
import { sequenceS } from 'fp-ts/Apply'
import * as O from 'fp-ts/Option'
import * as E from 'fp-ts/Either'

// Option: all fields must be Some for the result to be Some
const combined = sequenceS(O.Apply)({
  name: O.some('Alice'),
  age: O.some(30),
  city: O.some('Berlin'),
})
// O.some({ name: 'Alice', age: 30, city: 'Berlin' })

const withNone = sequenceS(O.Apply)({
  name: O.some('Alice'),
  age: O.none,              // <-- this makes everything None
  city: O.some('Berlin'),
})
// O.none — short-circuits on first None

// Either: all fields must be Right for the result to be Right
type FieldError = string

const validatedForm = sequenceS(E.Apply)({
  email: E.right('alice@example.com') as E.Either<FieldError, string>,
  name: E.right('Alice') as E.Either<FieldError, string>,
  age: E.right(30) as E.Either<FieldError, number>,
})
// E.right({ email: 'alice@example.com', name: 'Alice', age: 30 })

const invalidForm = sequenceS(E.Apply)({
  email: E.right('alice@example.com') as E.Either<FieldError, string>,
  name: E.left('Name is required') as E.Either<FieldError, string>,
  age: E.left('Age must be positive') as E.Either<FieldError, number>,
})
// E.left('Name is required') — Either short-circuits on first Left
// (For accumulating ALL errors, see the Validation section below)
```

### sequenceS with TaskEither — parallel async operations

```typescript
import { pipe } from 'fp-ts/function'
import { sequenceS } from 'fp-ts/Apply'
import * as TE from 'fp-ts/TaskEither'

interface User { id: string; name: string }
interface Activity { action: string; timestamp: Date }
interface NotificationCount { unread: number }

// These three operations are INDEPENDENT — no step needs the output of another
declare const fetchUser: (id: string) => TE.TaskEither<Error, User>
declare const fetchActivity: (id: string) => TE.TaskEither<Error, ReadonlyArray<Activity>>
declare const fetchNotificationCount: (id: string) => TE.TaskEither<Error, NotificationCount>

// ApplyPar runs all three Promises simultaneously (like Promise.all)
const loadDashboard = (userId: string) =>
  sequenceS(TE.ApplyPar)({
    user: fetchUser(userId),
    activity: fetchActivity(userId),
    notifications: fetchNotificationCount(userId),
  })
// TE.TaskEither<Error, { user: User, activity: Activity[], notifications: NotificationCount }>

// If any fails, the whole thing fails with that error
// On success, you get a nice typed record with all three results

// Usage
const result = await loadDashboard('user-123')()
// E.right({ user: { id: '...', name: 'Alice' }, activity: [...], notifications: { unread: 3 } })
```

### sequenceT — combining effects into a tuple

```typescript
import { sequenceT } from 'fp-ts/Apply'
import * as O from 'fp-ts/Option'
import * as TE from 'fp-ts/TaskEither'

// sequenceT produces a tuple instead of a named record
const tuple = sequenceT(O.Apply)(
  O.some('hello'),
  O.some(42),
  O.some(true)
)
// O.some(['hello', 42, true])

// Useful when you want to destructure the result
const [a, b, c] = O.getOrElse(() => ['', 0, false])(tuple)

// With TaskEither
const parallelPair = sequenceT(TE.ApplyPar)(
  fetchUser('123'),
  fetchActivity('123')
)
// TE.TaskEither<Error, [User, ReadonlyArray<Activity>]>
```

### Parallel validation — collecting ALL field errors

```typescript
import { pipe } from 'fp-ts/function'
import { sequenceS } from 'fp-ts/Apply'
import * as E from 'fp-ts/Either'
import * as NEA from 'fp-ts/NonEmptyArray'

// For accumulating errors, use Either's Applicative with a Semigroup for the error type
// The getApplicativeValidation helper creates an Apply instance that accumulates errors
import { getApplicativeValidation } from 'fp-ts/Either'
import { getSemigroup } from 'fp-ts/NonEmptyArray'

type ValidationError = NonEmptyArray<string>

const lift = <A>(e: E.Either<string, A>): E.Either<ValidationError, A> =>
  pipe(e, E.mapLeft((err) => [err] as ValidationError))

const validateEmail = (email: string): E.Either<ValidationError, string> =>
  lift(email.includes('@') ? E.right(email) : E.left('Invalid email'))

const validateName = (name: string): E.Either<ValidationError, string> =>
  lift(name.trim().length > 0 ? E.right(name.trim()) : E.left('Name required'))

const validateAge = (age: number): E.Either<ValidationError, number> =>
  lift(age >= 18 ? E.right(age) : E.left('Must be 18 or older'))

// Use getApplicativeValidation to ACCUMULATE errors instead of short-circuiting
const ApplicativeValidation = getApplicativeValidation(getSemigroup<string>())

const validateForm = (email: string, name: string, age: number) =>
  sequenceS(ApplicativeValidation)({
    email: validateEmail(email),
    name: validateName(name),
    age: validateAge(age),
  })

validateForm('', '', 15)
// E.left(['Invalid email', 'Name required', 'Must be 18 or older'])
// ALL three errors collected at once!

validateForm('alice@example.com', 'Alice', 25)
// E.right({ email: 'alice@example.com', name: 'Alice', age: 25 })
```

## When to Use / When NOT to Use

### Use sequenceS / Apply when:

- You have multiple **independent** computations whose results you need to combine
- You want to run multiple async operations in parallel (use `TE.ApplyPar`)
- You are validating multiple fields simultaneously and want all errors at once
- You need to build a record from multiple optional or fallible sources

### Do NOT use sequenceS when:

**Operations are dependent.** If step 2 needs the output of step 1, use `chain`. `sequenceS` cannot express dependencies — all inputs must be already computed when you call it.

```typescript
// WRONG: userId needed for fetchProfile but sequenceS can't express this
const wrong = sequenceS(TE.ApplyPar)({
  user: fetchUserByEmail(email),
  profile: fetchProfile(/* where does userId come from? */),
})

// RIGHT: chain expresses the dependency
const right = pipe(
  fetchUserByEmail(email),
  TE.chain((user) => fetchProfile(user.id))
)
```

**Error accumulation semantics differ.** By default, `sequenceS(E.Apply)` short-circuits on the first Left — same as `chain`. For true error accumulation, you need `getApplicativeValidation` as shown above. This is a subtle but important distinction.

**You only have one effect.** `sequenceS` is for combining multiple effects. If you only have one, use `map` or `chain`.

## Real-World Example

A form validation handler that collects all errors at once, plus a parallel data loader for an admin dashboard.

```typescript
import { pipe } from 'fp-ts/function'
import { sequenceS } from 'fp-ts/Apply'
import * as TE from 'fp-ts/TaskEither'
import * as E from 'fp-ts/Either'

// --- Parallel dashboard data loading ---

interface DashboardData {
  metrics: { totalUsers: number; activeSessions: number }
  recentOrders: ReadonlyArray<{ id: string; total: number }>
  systemStatus: { healthy: boolean; uptime: number }
}

const loadDashboardData = (): TE.TaskEither<Error, DashboardData> =>
  sequenceS(TE.ApplyPar)({
    metrics: TE.tryCatch(
      () => fetch('/api/metrics').then((r) => r.json()),
      (e) => e as Error
    ),
    recentOrders: TE.tryCatch(
      () => fetch('/api/orders?limit=10').then((r) => r.json()),
      (e) => e as Error
    ),
    systemStatus: TE.tryCatch(
      () => fetch('/api/status').then((r) => r.json()),
      (e) => e as Error
    ),
  })

// All three fetches start simultaneously. Total time = slowest request (not sum of all).

// --- Loading and rendering ---
const renderDashboard = async () => {
  const result = await loadDashboardData()()

  pipe(
    result,
    E.fold(
      (err) => showErrorBanner(`Failed to load dashboard: ${err.message}`),
      (data) => {
        renderMetrics(data.metrics)
        renderOrderTable(data.recentOrders)
        renderStatusBadge(data.systemStatus)
      }
    )
  )
}

// Placeholder render functions
declare function showErrorBanner(msg: string): void
declare function renderMetrics(m: { totalUsers: number; activeSessions: number }): void
declare function renderOrderTable(orders: ReadonlyArray<{ id: string; total: number }>): void
declare function renderStatusBadge(status: { healthy: boolean; uptime: number }): void
```

The parallel sequenceS approach is not just cleaner than chaining — it is measurably faster. Three independent 200ms requests take 200ms total with `ApplyPar`, but 600ms if you chain them sequentially.

## Key Takeaways

- **`sequenceS(Apply)({...})`** combines independent named effects into a single effect returning a record — use it when you have multiple unrelated computations to combine
- **`TE.ApplyPar`** runs TaskEither operations concurrently (like `Promise.all`); **`TE.ApplySeq`** runs them sequentially — prefer `ApplyPar` for truly independent operations
- **chain vs sequenceS**: chain is for sequential dependent steps; sequenceS is for parallel independent steps
- **Error accumulation** requires `getApplicativeValidation` — plain `sequenceS(E.Apply)` still short-circuits on the first error, same as chain
- **Apply is the missing link** between Functor (map over one effect) and Monad (chain dependent effects) — it handles the "independent parallel" case

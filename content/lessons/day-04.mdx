Every async operation in JavaScript returns a Promise — a value that represents work that will complete in the future. Promises are fine, but they fire the moment you create them and their error handling is a mix of `.catch()` chains and try/catch blocks. `Task` is a lazy Promise: it is a description of async work that has not started yet. You build up the pipeline first, then run it once at the end — giving you full control over when side effects happen.

## Introduction

Promises are the standard way to handle async code in TypeScript, and they work well for one-off operations. But they have a fundamental property that makes them awkward for functional composition: **Promises are eager**. When you create a `new Promise(...)` or call an `async` function, the async operation starts immediately. You cannot pass a Promise around as a value without the work already having started.

This eagerness causes several problems. A Promise cannot be retried — once it has settled, that is its final state. You cannot defer execution until some other condition is met. You cannot compose two "recipes for async work" without triggering the work immediately. And perhaps most importantly for functional programming, eager execution means Promises are not referentially transparent — a function that returns `fetch(url)` has a side effect the moment it is called, regardless of whether you use the returned Promise.

`Task<A>` solves this by representing an async computation as a **lazy function**: `() => Promise<A>`. The async work does not start until you call the function. This simple change — wrapping the Promise in a thunk — makes Task pure, composable, and easy to reason about. You can pass a Task around, combine it with other Tasks using `pipe`, and only trigger execution at the edge of your program when everything is wired up.

Task is intentionally simple. It models async operations that always succeed (never produce a typed error). For async operations that can fail, you will use `TaskEither` — covered in Day 5.

## Core API

### Creating Tasks

```typescript
import * as T from 'fp-ts/Task'

// T.of — lift a pure value into Task (the Task immediately resolves)
const immediateTask: T.Task<number> = T.of(42)
// Type: () => Promise<42>

// A custom async Task — just a function returning a Promise
const fetchUserId: T.Task<number> = () => Promise.resolve(42)

// An actual async operation
const getCurrentTime: T.Task<Date> = () => Promise.resolve(new Date())

// From an async function (async functions return Promises)
const readConfig: T.Task<string> = async () => {
  const response = await fetch('/api/config')
  return response.text()
}

// Running a Task: just call it
const result: Promise<number> = fetchUserId()
const value = await fetchUserId()  // 42
```

### Transforming Tasks with map and chain

```typescript
import { pipe } from 'fp-ts/function'
import * as T from 'fp-ts/Task'

// T.map — transform the value inside a Task (no async operation runs yet)
const fetchId: T.Task<number> = () => Promise.resolve(42)

const fetchAndDouble: T.Task<number> = pipe(
  fetchId,
  T.map((id) => id * 2)   // still lazy! no Promise has been created yet
)

// Nothing ran until here:
const result = await fetchAndDouble()   // 84

// T.chain — sequence Tasks where each step depends on the previous
const fetchUserName = (id: number): T.Task<string> =>
  () => Promise.resolve(`User #${id}`)

const fetchProfilePic = (name: string): T.Task<string> =>
  () => Promise.resolve(`https://avatars.example.com/${name.toLowerCase()}`)

// Sequential async pipeline — each step waits for the previous
const program: T.Task<string> = pipe(
  fetchId,                        // Task<number>
  T.chain(fetchUserName),         // Task<string> — runs after fetchId resolves
  T.chain(fetchProfilePic)        // Task<string> — runs after fetchUserName resolves
)

// Still nothing has run. Execute the whole pipeline:
const avatarUrl = await program()   // "https://avatars.example.com/user-#42"
```

### T.fromIO — lifting synchronous effects

```typescript
import { pipe } from 'fp-ts/function'
import * as T from 'fp-ts/Task'
import * as IO from 'fp-ts/IO'

// IO<A> = () => A — synchronous effect (covered in a later day)
const getTimestamp: IO.IO<number> = () => Date.now()

// Lift a synchronous IO into Task to mix with async operations
const timestampTask: T.Task<number> = T.fromIO(getTimestamp)

// Mix sync and async in one pipeline
const loggedOperation: T.Task<string> = pipe(
  timestampTask,
  T.chain((start) =>
    pipe(
      fetchId,
      T.map((id) => `[${start}] fetched id: ${id}`)
    )
  )
)
```

### Combining independent Tasks

```typescript
import { pipe } from 'fp-ts/function'
import * as T from 'fp-ts/Task'
import { sequenceT } from 'fp-ts/Apply'

// Run two independent Tasks in sequence (one waits for the other)
const fetchUser: T.Task<string> = () => Promise.resolve('Alice')
const fetchRole: T.Task<string> = () => Promise.resolve('admin')

// sequenceT(T.ApplyPar) — run in PARALLEL, collect results as a tuple
const parallel = sequenceT(T.ApplyPar)(fetchUser, fetchRole)
const [user, role] = await parallel()   // ['Alice', 'admin']
// Both Tasks started simultaneously

// sequenceT(T.ApplySeq) — run in sequence
const sequential = sequenceT(T.ApplySeq)(fetchUser, fetchRole)
const [u, r] = await sequential()   // ['Alice', 'admin']
// fetchUser ran first, then fetchRole
```

## When to Use / When NOT to Use

### Use Task when:

- You have an async operation that is guaranteed not to fail with a domain error (or you are handling errors inside the Task and always returning a value)
- You want to defer async work until your program is ready to execute it
- You need to compose multiple async operations sequentially with `T.chain`
- You want to pass async computations around as first-class values without triggering them

### Do NOT use Task when:

**The async operation can produce a typed error.** If the network request can fail, if the database query can return no results, if the file might not exist — use `TaskEither<E, A>` instead. `Task` assumes success and has no mechanism for error representation in the type.

**The operation is synchronous.** A pure synchronous computation should just be a function or use `IO<A>`. Wrapping a synchronous operation in `Task` adds unnecessary Promise indirection.

**You only need a one-shot operation.** For a single fetch call you are not composing with anything else, plain `async/await` is more readable and has less overhead. Task shines when you are building reusable, composable operations.

**You are using Task to delay something that should run eagerly.** Laziness is a tool, not a universal good. If you always call the Task immediately after creating it, you are getting no benefit from the wrapper — you might as well use a Promise or async function directly.

## Real-World Example

A realistic fetch pipeline: fetch a user's ID from a session, then fetch their profile, then transform the response into a UI model — all composed as a lazy pipeline executed once at startup.

```typescript
import { pipe } from 'fp-ts/function'
import * as T from 'fp-ts/Task'

interface SessionResponse {
  userId: string
  sessionToken: string
}

interface ProfileResponse {
  id: string
  displayName: string
  avatarUrl: string
  createdAt: string
}

interface UserViewModel {
  id: string
  name: string
  avatar: string
  memberSince: Date
}

// Individual Task operations — each is a lazy unit of async work
const fetchSession: T.Task<SessionResponse> = () =>
  fetch('/api/session').then((r) => r.json())

const fetchProfile = (userId: string): T.Task<ProfileResponse> => () =>
  fetch(`/api/users/${userId}`).then((r) => r.json())

const toViewModel = (profile: ProfileResponse): UserViewModel => ({
  id: profile.id,
  name: profile.displayName,
  avatar: profile.avatarUrl,
  memberSince: new Date(profile.createdAt),
})

// Compose the full pipeline — nothing runs yet
const loadCurrentUser: T.Task<UserViewModel> = pipe(
  fetchSession,
  T.chain((session) => fetchProfile(session.userId)),
  T.map(toViewModel)
)

// Execute the pipeline — this is where the async work actually happens
const initApp = async () => {
  const user = await loadCurrentUser()
  console.log(`Welcome, ${user.name}!`)
  renderApp(user)
}

function renderApp(user: UserViewModel) {
  // ... render the UI
}
```

Notice how `loadCurrentUser` is a value — a description of the computation, not the computation itself. You can pass it to other functions, compose it into larger pipelines, or call it multiple times. Each call starts fresh with a new chain of Promises.

This is the architectural advantage of Task: it lets you build up complex async workflows as composable, reusable values, and defer execution to the entry point of your program — keeping your business logic pure and testable.

## Key Takeaways

- **`Task<A>` is `() => Promise<A>`** — the laziness is the entire point; async work does not start until you call the function
- **`T.map`** transforms the resolved value without triggering execution; **`T.chain`** sequences dependent async operations
- **Task assumes success** — use `TaskEither<E, A>` when the async operation can fail with a typed error
- **Execute at the edge** — compose your Task pipeline throughout your program, then call it once at the entry point; this keeps your domain logic pure and side-effect-free
- **Task is a value** — pass it to functions, store it in variables, combine it with other Tasks before deciding when to run it
